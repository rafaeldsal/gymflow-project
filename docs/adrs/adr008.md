# ADR-008 — Versionamento do Banco de Dados com Flyway

## Status: Ativo

## Data: 09/07/2026

## Última atualização: 09/07/2026

## Contexto

Após definir o PostgreSQL como Sistema Gerenciador de Banco de Dados e adotar a estratégia de Multi-Tenant baseada em um schema por Tenant, tornou-se necessário estabelecer uma estratégia para controle da evolução do esquema do banco de dados.

À medida que novas funcionalidades forem adicionadas ao GymFlow, alterações estruturais serão inevitáveis, como criação de tabelas, inclusão de colunas, índices, constraints e demais objetos do banco de dados.

Realizar essas alterações manualmente aumentaria o risco de inconsistências entre ambientes e dificultaria a reprodução da estrutura do banco ao longo do ciclo de desenvolvimento.

## Decisão

Foi adotado o Flyway como ferramenta oficial para versionamento e migração do banco de dados do GymFlow.

Toda alteração estrutural deverá ser realizada por meio de scripts versionados, garantindo rastreabilidade, reprodutibilidade e consistência entre os diferentes ambientes da aplicação.

O Flyway será responsável por aplicar automaticamente as migrações durante a inicialização da aplicação, respeitando a estratégia de Multi-Tenant definida pelo projeto.

## Motivação

A adoção do Flyway foi motivada pelos seguintes fatores:

- Garantir que a estrutura do banco seja controlada por versionamento.
- Eliminar alterações manuais no banco de dados.
- Facilitar a criação de novos ambientes.
- Automatizar a evolução do esquema dos Tenants.
- Integrar naturalmente com Spring Boot e PostgreSQL.
- Aproximar o projeto das práticas utilizadas em aplicações corporativas.

Além dos benefícios técnicos, a utilização do Flyway proporciona experiência prática com uma ferramenta amplamente utilizada em projetos Java de médio e grande porte.

## Diretrizes

Como consequência desta decisão:

- Toda alteração estrutural deverá ser realizada por meio de migrations versionadas.
- Alterações manuais diretamente no banco deverão ser evitadas.
- O histórico das migrações fará parte do repositório do projeto.
- As migrations deverão ser executadas automaticamente durante a inicialização da aplicação.
- A estratégia de migração deverá respeitar a arquitetura Multi-Tenant adotada pelo GymFlow.

O versionamento da estrutura do banco será considerado parte integrante do código-fonte da aplicação.

## Consequências

### Positivas

- Histórico completo da evolução do banco de dados.
- Reprodutibilidade entre ambientes de desenvolvimento, homologação e produção.
- Maior segurança durante alterações estruturais.
- Facilidade para criação de novos Tenants.
- Automatização das migrações.

### Negativas

- Necessidade de manter disciplina na criação das migrations.
- Correções em migrations já executadas exigem novas versões, não alterações nos scripts existentes.
- Maior planejamento para alterações estruturais complexas.

## Alternativas Consideradas

### Hibernate DDL Auto (ddl-auto)

Foi considerado utilizar o mecanismo de geração automática de esquema disponibilizado pelo Hibernate.

Essa alternativa foi descartada por não fornecer controle de versão, histórico das alterações nem rastreabilidade das mudanças realizadas na estrutura do banco.

### Liquibase

Também foi considerada a utilização do Liquibase.

Embora seja uma ferramenta madura e amplamente utilizada, o Flyway apresentou uma abordagem mais simples para o contexto do GymFlow, com menor curva de aprendizado e excelente integração com o ecossistema Spring Boot.

## Revisão Futura

O Flyway permanecerá como ferramenta oficial de versionamento do banco enquanto atender às necessidades do projeto.

Caso a estratégia de migração evolua para cenários mais complexos, envolvendo múltiplas bases de dados ou requisitos específicos de governança, esta decisão poderá ser revisada.

Independentemente da ferramenta utilizada, o princípio de manter a estrutura do banco de dados versionada continuará sendo obrigatório no GymFlow.
