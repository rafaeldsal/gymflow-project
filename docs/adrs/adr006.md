# ADR-006 - Estratégia de Multi-Tenant

## Status: Ativo

## Data: 09/07/2026

## Última atualização: 09/07/2026

## Contexto

O GymFlow foi concebido desde o início como uma solução SaaS (Software as a Service), onde uma única instância da aplicação será responsável por atender múltiplas academias.

Cada academia representa um cliente (Tenant) da plataforma e possui seus próprios alunos, professores, planos, matrículas e demais informações.

Diante desse cenário, tornou-se necessário definir uma estratégia de isolamento dos dados, garantindo que informações pertencentes a um Tenant nunca sejam acessadas por outro.

Além do isolamento, a solução escolhida deveria equilibrar simplicidade de implementação, baixo custo operacional e possibilidade de evolução futura.

## Decisão

Foi adotada a estratégia de Multi-Tenant por Schema, utilizando um schema independente para cada Tenant dentro da mesma instância do banco de dados.

Todos os módulos da aplicação compartilharão essa estratégia de persistência.

A identificação do Tenant ocorrerá durante o processamento de cada requisição e será utilizada para selecionar automaticamente o schema correspondente antes da execução das operações de persistência.

A responsabilidade por essa resolução ficará centralizada na camada de infraestrutura da aplicação.

## Motivação

A escolha da estratégia Schema per Tenant foi motivada pelos seguintes fatores:

- Garantir isolamento lógico entre os dados de cada academia.
- Facilitar o gerenciamento da infraestrutura utilizando uma única instância do banco de dados.
- Reduzir custos operacionais durante as fases iniciais do projeto.
- Simplificar processos de backup e restauração por Tenant.
- Permitir evolução futura para outras estratégias de Multi-Tenant caso necessário.
- Proporcionar experiência prática com uma arquitetura SaaS amplamente utilizada no mercado.

Além dos aspectos técnicos, esta decisão está alinhada ao objetivo de estudar os desafios envolvidos na construção de aplicações multi-tenant utilizando o ecossistema Spring.

## Diretrizes

Como consequência desta decisão:

- Cada Tenant possuirá um schema exclusivo.
- Todas as tabelas de domínio serão criadas dentro do schema do respectivo Tenant.
- O contexto do Tenant será resolvido antes de qualquer acesso à persistência.
- A infraestrutura será responsável por alterar automaticamente o schema ativo durante o ciclo de vida da requisição.
- Os módulos de negócio não deverão possuir conhecimento sobre a estratégia de Multi-Tenant adotada.

O domínio continuará trabalhando apenas com seus conceitos de negócio, permanecendo desacoplado dos mecanismos utilizados para isolamento dos dados.

## Consequências

### Positivas

- Isolamento lógico entre academias.
- Menor custo operacional quando comparado a um banco por Tenant.
- Facilidade para gerenciamento das migrações de banco.
- Boa escalabilidade para pequenas e médias quantidades de clientes.
- Transparência para os módulos de negócio.

### Negativas

- Maior complexidade na configuração da infraestrutura de persistência.
- Necessidade de gerenciamento do ciclo de vida dos schemas.
- Migrações deverão ser executadas para todos os Tenants existentes.
- Crescimento elevado do número de schemas conforme a plataforma evoluir.

## Alternativas Consideradas

### Banco de Dados por Tenant

Oferece maior isolamento físico entre clientes.

Entretanto, aumenta significativamente o custo operacional e a complexidade de gerenciamento da infraestrutura, especialmente durante as fases iniciais do projeto.

### Coluna Identificadora (Shared Database)

Consiste em armazenar todos os dados na mesma tabela utilizando uma coluna de identificação do Tenant.

Embora seja uma abordagem simples de implementar, foi descartada por aumentar o risco de vazamento de dados entre clientes e exigir maior atenção durante o desenvolvimento de consultas.

### Schema por Tenant

Foi escolhida por oferecer um bom equilíbrio entre isolamento, simplicidade operacional e escalabilidade para o contexto atual do GymFlow.

## Revisão Futura

A estratégia de Multi-Tenant deverá ser reavaliada caso o crescimento da plataforma justifique mudanças na infraestrutura.

Caso requisitos de escalabilidade, isolamento físico ou regulamentações específicas tornem necessário um nível maior de segregação, a arquitetura poderá evoluir para uma estratégia baseada em banco de dados por Tenant.

Até que esse cenário se concretize, a estratégia de Schema por Tenant permanecerá como a abordagem oficial do GymFlow.
