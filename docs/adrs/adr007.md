# ADR-007 — Adoção do PostgreSQL como Sistema Gerenciador de Banco de Dados

## Status: Ativo

## Data: 09/07/2026

## Última atualização: 09/07/2026

## Contexto

Após definir a estratégia de Multi-Tenant baseada em um schema por Tenant, tornou-se necessário selecionar um Sistema Gerenciador de Banco de Dados (SGBD) capaz de atender aos requisitos arquiteturais do GymFlow.

A solução deveria oferecer suporte sólido ao modelo relacional, recursos para gerenciamento de múltiplos schemas, integração com o ecossistema Spring e boa aderência a aplicações SaaS.

Além disso, a tecnologia escolhida deveria representar uma ferramenta amplamente utilizada no mercado, permitindo seu uso como parte da trilha de aprendizado proposta para o projeto.

## Decisão

Foi adotado o PostgreSQL como Sistema Gerenciador de Banco de Dados oficial do GymFlow.

O PostgreSQL será responsável pelo armazenamento das informações transacionais da aplicação e servirá como base para a implementação da estratégia de Multi-Tenant definida no ADR-006.

A configuração do banco será centralizada no módulo de infraestrutura de persistência, permanecendo transparente para os módulos de negócio.

## Motivação

A escolha do PostgreSQL foi motivada pelos seguintes fatores:

- Suporte nativo ao uso de múltiplos schemas.
- Excelente integração com Spring Boot, Spring Data JPA e Hibernate.
- Alto nível de conformidade com o padrão SQL.
- Ampla adoção em aplicações corporativas e plataformas SaaS.
- Ecossistema maduro e documentação abrangente.
- Licenciamento open source.
- Oportunidade de aprofundar conhecimentos em um dos bancos de dados relacionais mais utilizados no mercado.

## Diretrizes

Como consequência desta decisão:

- O PostgreSQL será o banco de dados relacional oficial do projeto.
- A persistência será realizada por meio do Spring Data JPA e Hibernate.
- A criação e evolução do esquema do banco serão controladas por ferramentas de migração versionada.
- Cada Tenant possuirá seu próprio schema dentro da mesma instância do PostgreSQL.
- Os módulos de domínio permanecerão independentes da tecnologia utilizada para persistência.

## Consequências

### Positivas

- Excelente suporte ao modelo de Multi-Tenant adotado.
- Alta confiabilidade para aplicações transacionais.
- Boa performance em consultas relacionais.
- Facilidade de integração com o ecossistema Java.
- Possibilidade de utilizar recursos avançados do PostgreSQL conforme o projeto evoluir.

### Negativas

- Dependência de um banco relacional para o armazenamento principal da aplicação.
- Necessidade de conhecimento específico para administração e otimização do PostgreSQL.
- Recursos específicos do PostgreSQL podem reduzir a portabilidade para outros SGBDs caso sejam utilizados de forma intensiva.

## Alternativas Consideradas

### MySQL

Foi considerado devido à sua ampla adoção no mercado.

Entretanto, o suporte do PostgreSQL ao gerenciamento de múltiplos schemas e sua aderência a aplicações SaaS o tornaram mais adequado para os objetivos do GymFlow.

### MariaDB

Também foi avaliado por ser compatível com diversas ferramentas do ecossistema MySQL.

Apesar disso, o PostgreSQL apresentou melhor alinhamento com os requisitos arquiteturais definidos para o projeto.

### Microsoft SQL Server

Foi descartado por possuir um ecossistema mais voltado para ambientes corporativos específicos e por não atender ao objetivo de construir uma solução baseada em tecnologias open source.

## Revisão Futura

O PostgreSQL permanecerá como banco de dados principal enquanto atender aos requisitos funcionais e não funcionais do GymFlow.

Caso novos requisitos justifiquem o uso de tecnologias especializadas, como bancos NoSQL para casos específicos, essas soluções poderão ser incorporadas de forma complementar, preservando o PostgreSQL como principal repositório transacional da aplicação.
